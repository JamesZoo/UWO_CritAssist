## What this project is

iOS app that helps a user iteratively improve a recipe using Apple
Intelligence on-device. Built on iPad in Swift Playgrounds because the
user has **no Mac**.

Two primary surfaces:

1. **Add a recipe** by dish name, paste, or URL. AI generates / parses /
   cleans the recipe and fetches a representative photo (Wikipedia real
   photo preferred, ImagePlayground-generated fallback).
2. **Refine** by giving feedback. AI diagnoses why the feedback happened
   and proposes targeted changes. User approves or discards. Rollback
   per revision is supported.

Plus: branched variations, final analysis (best-of-base + best-of-each-
variation document), step-by-step AI-generated illustrations on key
cooking moments.

## Honest known limitations

These are real constraints, not "todo soon":

- **No direct image input to `LanguageModelSession`** in the current
  FoundationModels API. We probed it (`Vision/01-DirectImageInput`) and
  the `image:` parameter doesn't exist. Image validation today is
  text-based ŌĆö we compare a Wikipedia article title to the dish name
  using AI judgment of visual similarity. `Vision.VNClassifyImageRequest`
  works (probe 2) but its labels are too generic for dish recognition
  (probe 3 confirmed it can't distinguish sushi from pizza).
- **ImagePlayground style**: only `.animation` is confirmed working.
  Output is stylized cartoon, not photorealistic. For "real photos
  generated by AI" we'd need either a different style (untested) or an
  external API (would require paid keys; deferred).
- **Apple Intelligence safety filter** is aggressive on Chinese dish
  names (`ńłåńéÆĶģ░ĶŖ▒`, `ń║óńā¦µÄÆķ¬©`, others). Mitigations in place: catalog-
  lookup framing + three-attempt retry (English, English-simpler,
  Chinese-only) + a clear "safety filter declined" banner with fallback
  to Paste mode. See `AppleIntelligenceServices.swift`.
- **Language drift**: the model sometimes outputs English for Chinese
  dish names despite explicit prompt directives. Post-generation
  language enforcement (`enforceLanguage` methods on the generator and
  refiner) detects the mismatch via `LanguageHeuristics` and runs a
  translation pass. This is the working safety net.
- **File path stability**: AI-generated image URLs are saved as
  absolute paths that include the app container UUID. The UUID can
  change between launches in Swift Playgrounds. `LocalImagePathResolver`
  re-roots saved file URLs at display time. Don't bypass it.
- **iCloud sync, CI/TestFlight, on-device tests**: all deferred. Tests
  exist (`Tests/AppModuleTests/`) but only as `swift test` artifacts;
  Swift Playgrounds doesn't run them. The test target was excluded from
  the Package.swift back in commit `93d8591` because Playgrounds can't
  link the Swift Testing framework.

## Communication notes for the assistant

- The user prefers tight, factual responses. No emojis. Short paragraphs.
  Name files and line numbers when relevant.
- The user is bilingual (English + Chinese). They occasionally write in
  Chinese; match their language when replying.
- When the user reports a bug, **read the screenshot carefully** before
  proposing a fix. They've caught me twice for misreading errors.
- When you push a commit, tell the user the SHA so they know what to
  pull. Pull-and-test is the only feedback loop available.

## Anti-patterns to avoid

- **Don't write speculative AI service code without probes**. The
  `1067f4b ŌåÆ 3e7d163` revert cycle was costly to the user's time.
- **Don't bypass `LocalImagePathResolver`** when displaying images. Use
  it in every new view that shows an AI-generated thumbnail.
- **Don't add `+`-chained string concatenations with `.map().joined()`
  in the same expression** ŌĆö Swift's type checker times out. Use string
  interpolation with local lets instead. Two places have already been
  fixed (`enforceLanguage` in generator + refiner).
- **Don't widen the test target** in `Package.swift` without re-verifying
  Swift Playgrounds tolerates it. Currently no test target ŌĆö added back
  by CI later.
- **Don't claim a feature is shipped** when the underlying API has only
  been guessed at. Add a probe first, confirm, then implement.
